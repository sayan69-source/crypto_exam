CryptoExam Core replaces human trust with mathematical enforcement at every layer.

## 3. The Cascading Trust & Registration Chain

In standard systems, anyone with an "Admin" password can access the system. In CryptoExam, trust must be cryptographically handed down from the very top. No tier can approve itself.

### Tier 0: System Admin (Headquarters)
- **Registration:** The System Admin is not registered through a web form. Their identity and authority are established during an offline "key ceremony" where the master encryption keys are generated. The private key to decrypt answers is placed inside a Hardware Security Module (HSM) at HQ.
- **Role:** They are the absolute root of trust. They monitor nationwide operations, anchor hashes to the Polygon blockchain, and are the *only* entity capable of decrypting answers.

### Tier 1: Centre Admin (Exam Centre)
- **Registration Flow:** A designated person goes to the public frontend portal and registers to be a Centre Admin for a specific location. Their registration is marked as `PENDING_APPROVAL`.
- **Approval:** This pending request appears on the System Admin's dashboard at HQ. To approve it, the System Admin physically verifies the person's identity and generates a cryptographically random, single-use, short-lived **One-Time Code**. The System Admin also authorises the binding of the Centre Admin's fingerprint.
- **Activation:** The Centre Admin enters this code at their station. The system verifies the code, binds their face and fingerprint templates, and sets their status to `ACTIVE`. Only one active Centre Admin is allowed per centre.
- **Database Constraint:** Enforced via:
  ```sql
  CREATE UNIQUE INDEX one_active_center_admin ON staff_identities(center_id) WHERE role='CENTER_ADMIN' AND status='ACTIVE';
  ```

### Tier 2: Centre Invigilator (Exam Centre Floor)
- **Registration Flow:** The staff member uses an Invigilator Station at the centre, captures their face and fingerprint, and submits a `PENDING` registration.
- **Approval:** This request surfaces on the Centre Admin's local, air-gapped portal. The Centre Admin visually verifies the staff member, generates a one-time code, and clicks "Authorise & bind fingerprint".
- **Activation:** The staff member enters the code. They are now an `ACTIVE` Invigilator.

### The "Match-All" Login Sequence
When an Invigilator or Centre Admin attempts to log in, the Edge Server enforces a strict "Match-All" intersection rule. The login succeeds **IF AND ONLY IF**:
```python
privileged_login_ok :=
      face_match(score >= 0.82)
  AND fingerprint_match(score >= vendor_threshold)
  AND source_ip == bound_ip(identity)
  AND tpm_attestation_valid(terminal)
  AND identity.status == ACTIVE
  AND not identity.revoked
  AND within_login_time_box(<= 20 s from challenge)
```
If *any* of these fail, the login is instantly denied, the terminal locks, and an incident is logged.

## 4. ZUUP-OS: The Custom Air-Gapped Operating System

Commodity PCs at the exam centre run ZUUP-OS to eliminate hardware and OS-level tampering.

### Boot Sequence & Hardware Trust (L0-L1)
- **UEFI Secure Boot:** The system locks the UEFI setup with a password and disables boot from USB/Optical. Only the kernel and bootloader signed by the exam authority's keys can boot.
- **PXE Chainloading:** The terminal boots over the network (PXE) from the Centre Edge Server, loading a ~300 MB SquashFS image into RAM.
- **Measured Boot & TPM Attestation:** On every boot, the TPM 2.0 provides remote attestation to the Edge Server. The Edge holds the golden PCR set. A mismatch halts the terminal at a breach screen.

### The OS Lockdown Stack (L2-L5)
- **Filesystem (L2):** SquashFS provides a read-only root. OverlayFS mounts `tmpfs` over `/var/log`, `/tmp`, and `/run`. All writes happen in RAM.
- **Network (L3):** Completely air-gapped. Only wired Ethernet is compiled in the kernel. WireGuard mTLS connects strictly to the Edge IP.
- **Display Lock (L4):** Wayland + Cage single-surface compositor. Firefox ESR is launched `--kiosk`.
- **Runtime Enforcement (L5):** Cilium Tetragon (eBPF) watches every syscall. The allowed process set is strictly limited (`systemd`, `cage`, `firefox`, `wg`, `biometric-daemon`).

### Firewalld & nftables Configuration

### The iON/iLeon Hand-off & Seat Assignment
1. **Physical Check-in:** Invigilator scans candidate face and fingerprint.
2. **Atomic Random Assignment:** 
```sql
SELECT * FROM terminals
WHERE centre_id = :c AND capability='CANDIDATE_SEAT'
  AND state='AVAILABLE' AND health='OK'
ORDER BY random() LIMIT 1
FOR UPDATE SKIP LOCKED;
```
3. **The Lock-in:** The computer instantly refreshes and displays the Candidate Login page, strictly bound to that single Roll Number.
4. **Login:** Candidate logs in using DOB as a password.

### Lazy Decryption (Last-Moment Question Reveal)
Questions are pre-downloaded but encrypted. At exactly `T0`, a cryptographic randomness beacon (`drand`) releases randomness used to derive AES decryption keys locally in RAM. Questions decrypt dynamically on a click-by-click basis, mimicking TCS-iON without relying on a central database connection.

## 6. Answer Egress: Hash Compilation & The "Blockchain" Path

### Step 1: The Encrypted Envelope
When submitted, a JSON record (`R`) is formed:
```json
{
  "exam_id": "uuid",
  "subject_ref": "pseudonymous_candidate_id",
  "responses": [
    {
      "question_hash": "0x...",
      "chosen_option": "B",
      "answered_at_ms": 123456789
    }
  ]
}
```
`R` is encrypted with a random AES-256 Data Key (`DK`). The `DK` is wrapped with the System Admin's RSA Public Key.

### Step 2: The Merkle Hash-Chain
The Edge Server hashes the encrypted package (`Leaf`) and appends it to a local Merkle Hash-Chain. The TPM signs the chain `Root`. The candidate receives a cryptographic receipt containing their `Leaf` and the `Root`. The Centre Admin stores only this ciphertext.

### Step 3: Handover and Final Decryption
```text
   CANDIDATE SEAT (RAM only)                CENTRE EDGE / CENTRE ADMIN STORE
   ─────────────────────────                ──────────────────────────────────
   build R  ──▶ DK=rand                      append leaf_n → root_n
   ct=AESGCM(DK,R)                           sig_n = TPM_sign(root_n)
   wrapped_DK=RSA-OAEP(SA_pub,DK)  ───push──▶ store {ct,iv,tag,wrapped_DK,proof}
   leaf=SHA256(ct…)                          (CANNOT DECRYPT — no SA private key)
   show receipt ◀──proof/root/sig──────────  │
                                             │  out-of-band, after window,
                                             ▼  via System Admin uplink (not live)
                                   ┌─────────────────────────────────────┐
                                   │ SYSTEM ADMIN @ HQ (HSM)             │
                                   │ 1. verify chain + node sig          │
                                   │ 2. anchor root_n on Polygon         │
                                   │ 3. DK = HSM_unwrap(wrapped_DK)      │
                                   │ 4. R = AESGCM_open(DK, ct)          │
                                   │ 5. store plaintext in SA DB ✅      │
                                   └─────────────────────────────────────┘
```
The database is out-of-band synced to HQ. The System Admin unwraps the `DK`s using an HSM. Finally, the final Merkle Root is anchored to the Polygon PoS blockchain. The examining agency gets the decrypted text, and the public can verify the Polygon root to prove no tampering occurred.

## 7. The AI Agent Pipeline & Quality Control

Generating thousands of questions securely and fairly requires an advanced AI orchestration pipeline. We utilize 6 specialized agents operating sequentially:

```text
┌──────────────┐     ┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  Generator   │────▶│  IRT Scorer  │────▶│   Bloom's    │────▶│  Validator   │
│  Agent       │     │  Agent       │     │   Agent      │     │  Agent       │
│ Instructor + │     │ 3PL params   │     │ L1-6 keyword │     │ Accept if    │
│ OpenAI/Mock  │     │ b/a/c        │     │ EN + HI      │     │ IRT ∈ range  │
└──────────────┘     └──────────────┘     └──────────────┘     └──────┬───────┘
                                                                      │
                                                              ┌───────▼───────┐
                                                              │   Balancer    │
                                                              │   Agent       │
                                                              │ Set A/B/C/D   │
                                                              │ equivalence   │
                                                              └───────────────┘
```
- **Generator Agent:** Utilizes large language models mapped via the `Instructor` library strictly to JSON schemas to generate diverse, context-accurate question sets.
- **IRT Scorer:** Calibrates questions using Item Response Theory. Specifically, the 3-Parameter Logistic (3PL) model calculates:
  - *a (Discrimination):* How well the question separates high-ability students from low-ability students.
  - *b (Difficulty):* The overall cognitive load of the question.
  - *c (Pseudo-guessing):* The probability that a student could guess the correct answer without knowing the subject.
- **Bloom's Agent:** Assesses cognitive levels. It parses verbs to categorize questions into Remembering, Understanding, Applying, Analyzing, Evaluating, or Creating.
- **Validator Agent:** Acts as an automated quality control gate. It rejects any question that violates formatting guidelines or falls outside acceptable IRT ranges.
- **Balancer Agent:** Ensures that different variants of the test (Sets A, B, C, and D) are perfectly statistically equivalent, preventing any "set-advantage" cheating.

## 13. Data Privacy (DPDP Act 2023) Compliance Specifics

The platform is built from the schema level up to comply with India's DPDP (Digital Personal Data Protection) Act 2023.

- **Section 4 (Consent):** Explicit consent flow is captured with IP address, timestamp, and terms version.
- **Section 8 (Security):** AES-GCM-256 encryption, TPM 2.0 remote attestation, and zero plaintext on disk ensure compliance with strict data security mandates.
- **Section 9 (Minors):** Mandatory parental consent gates are implemented for candidates under 18 years of age.
- **Section 16 (Data Rights):** Built-in API endpoints for data access, correction, and erasure.
- **No PII on Blockchain:** Only cryptographic hashes (never names, Aadhaar numbers, or answers) are committed to the Polygon blockchain.
- **Biometric Handling:** Raw biometric data is never stored. Only the mathematical embedding hash of facial features and minutiae templates of fingerprints are stored.

## 18. WireGuard Peer Configuration Pattern
Every ZUUP-OS terminal is a WireGuard peer strictly communicating with the Edge server.
```ini
[Interface]
PrivateKey = <Pre-baked Secure Boot Key>
Address = 10.10.x.x/32
ListenPort = 51820

[Peer]
PublicKey = <Edge Server Public Key>
AllowedIPs = 10.10.0.1/32
Endpoint = <Edge Server LAN IP>:51820
PersistentKeepalive = 25
```
By restricting `AllowedIPs` strictly to the Edge Server's single IP address (`10.10.0.1/32`), the terminal physically cannot route traffic anywhere else. The Linux kernel's routing table simply drops any packet destined for the public internet. This creates an impenetrable air-gap, making remote code execution and data exfiltration impossible, even if an attacker were to bypass Firefox and gain a shell.

## 19. Disaster Recovery & Emergency Protocols
In the event of a catastrophic failure, such as a localized power outage or Edge Server hardware failure, the system is designed to fail gracefully.
- **Fail-Closed Principle:** Terminals will instantly lock if the heartbeat to the Edge Server is lost. 
- **Offline Sync Preservation:** Any answers submitted prior to the failure are safely persisted in the Edge Server's PostgreSQL database. If the Edge Server hardware fails, the physical drives can be securely transported to HQ, where the System Admin can manually extract the ciphertext.
- **Emergency Pause:** Both Centre Admins (locally) and System Admins (nationally) can trigger an `EMERGENCY_PAUSE`. This instantly locks all active terminals and prevents any new submissions or logins until the pause is lifted and a new one-time code is issued.
